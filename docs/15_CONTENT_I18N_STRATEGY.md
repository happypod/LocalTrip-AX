# Content Internationalization (i18n) Strategy

이 문서는 LocalTrip AX의 운영 콘텐츠(숙소, 체험, 주민소득상품, 여정의 기록 등)에 대한 다국어(영어, 중국어, 일본어) 적용 전략과 운영 기준을 정의합니다. (T-047 기준 작성)

## 1. 콘텐츠 다국어 적용 필요성
현재 T-044를 통해 시스템 UI(네비게이션, 푸터, 주요 배너 문구 등)는 사전 정의된 Locale Dictionary를 통해 다국어를 완벽히 지원하고 있습니다. 하지만 DB에 저장되어 관리자가 수시로 등록/수정하는 **운영 콘텐츠**의 다국어 관리는 운영 리소스와 확장성을 고려해 신중한 접근이 필요합니다.
- **외국인 타겟팅**: 글로벌 마케팅을 위해서는 핵심 콘텐츠의 외국어 번역이 필수적입니다.
- **운영자 리소스 제한**: 모든 콘텐츠를 4개 국어로 실시간 번역하여 등록하는 것은 MVP 초기 운영 단계에서 현실적으로 불가능합니다.
- **번역 품질 보장**: 기계 번역과 전문가 번역을 언제 어떻게 결합할 것인지의 정책이 필요합니다.

## 2. 다국어 관리 방식 비교

### 방식 A: 한국어 원문만 유지 (현재 상태 유지)
- **특징**: DB 변경 없음, UI Dictionary만 다국어 지원.
- **장점**: 개발/운영 비용이 전혀 들지 않음. MVP의 빠른 출시에 유리함.
- **단점**: 사용자가 브라우저 자체 번역 기능에 의존해야 하므로, 매끄러운 다국어 UX 제공에는 한계가 있음.

### 방식 B: 각 모델에 JSON 필드 추가 (`translations Json?`)
- **특징**: `Accommodation`, `Experience` 등 개별 모델에 `translations` 필드 추가.
- **장점**: 모델 스키마의 수정이 적고 직관적인 조회가 가능함.
- **단점**: Prisma의 JSON 필드 내부에 대한 타입 안정성이 부족하며, 언어별 검색/필터링 조건 추가 시 쿼리 복잡도가 급격히 상승함.

### 방식 C: Translation 테이블 분리 (`ContentTranslation`)
- **특징**: 대상 타입(`targetType`)과 대상 ID(`targetId`)를 키로 가지는 별도의 다국어 번역 전용 테이블 신설.
- **장점**: 스키마 확장이 매우 용이하며, 번역 데이터에 대한 텍스트 검색, 개별 언어별 캐싱, 번역 상태(초안/완료) 관리에 최적화된 엔터프라이즈급 구조.
- **단점**: 조회 시 `JOIN` 비용이 발생하며, 관리자 및 클라이언트의 구조적 변경 폭이 가장 큼.

## 3. 권장안 (Recommendation)

LocalTrip AX의 비즈니스 성장 단계를 고려하여 단계적으로 도입합니다.

- **단기 (MVP 단계)**: **방식 A (한국어 원문 유지)**
  - UI 껍데기만 다국어 지원. 콘텐츠는 한국어로 노출되며 브라우저 번역기에 의존. 
- **외부 공개 및 마케팅 단계**: **방식 A + 수동 사전 번역 (일부)**
  - 외국인 타겟팅이 확실한 소수 핵심 상품에 한정하여 한국어 본문 내 영문 병기 혹은 단순화.
- **확장 단계 (Post-MVP)**: **방식 C (`ContentTranslation` 테이블 분리)**
  - 체계적인 다국어 지원이 필요한 시점에 Translation 전용 테이블을 구축.
  - 이 단계에서 AI 기반 자동 번역(초안 생성) 기능을 관리자에 연동.

## 4. Fallback 정책

방식 C(Translation 테이블) 도입 후 다국어 콘텐츠 렌더링 시 적용할 Fallback(대체) 순서입니다.

1. **선택 언어 콘텐츠**: 유저가 선택한 언어(예: 중국어)의 번역본이 존재하면 최우선 노출.
2. **영어 콘텐츠 (글로벌 기본어)**: 선택 언어 번역본이 없을 시, 가장 보편적인 영어 번역본 노출 시도.
3. **한국어 원문**: 영어 번역본마저 없을 시, 반드시 한국어 원문을 노출하여 빈 공간을 방지함.

**[주의 사항]**
- 콘텐츠 제목은 어떠한 경우에도 빈 값으로 노출되지 않아야 합니다.
- 시스템 안내 문구(CTA 버튼, 뱃지 등)는 기존 Dictionary 문구를 그대로 유지합니다.
- 사용자가 선택한 언어와 다른 언어(Fallback)가 노출될 경우, 이를 자동 번역된 것으로 위장하지 않습니다.

## 5. 관리자 UX 기준

추후 다국어 번역 시스템이 도입될 경우, 관리자 입력 인터페이스는 다음 기준을 따릅니다.

- **탭 기반 인터페이스**: 기본 탭은 `한국어(KR)`로 고정. 우측으로 `EN`, `CN`, `JP` 탭을 배치하여 직관적 전환 지원.
- **번역 대상 필드**: 언어별로 `Title`(제목), `Summary`(요약), `Description`(상세 설명)만 우선 번역 대상으로 개방. (Price, Address 등은 글로벌 공통 포맷 사용)
- **번역 대상 모델 허용값 (Allowlist)**: `ContentTranslation.targetType`은 String 타입이므로 코드 레벨 검증이 필수적입니다. 허용값: `accommodation`, `experience`, `local_income_program`, `course`, `event`, `training_course`, `certification`.
- **유효성 검사**: 외국어 탭의 필드가 비어 있더라도 저장이 가능해야 하며(Draft 상태), 빈 필드는 공개 화면에서 한국어 원문으로 Fallback 처리됨.
- **AI 어시스턴트 (후속)**: 각 외국어 탭 진입 시 "한국어 원문 기반 AI 번역 초안 생성" 버튼을 제공하여 운영자 편의성을 극대화.
