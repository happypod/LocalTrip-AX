# 관광객 맞춤코스 및 사용자 대시보드 확장 명세

작성일: 2026-05-14

## 1. 현재 상태

현재 LocalTrip AX는 운영자 중심 MVP다.

- 관리자 대시보드: `/admin`
- 공개 화면: `/`, `/stays`, `/experiences`, `/programs`, `/courses`, `/events`, `/map`
- 찜 기능: 브라우저 `localStorage` 기반 `WishlistProvider`
- 관광객 회원가입/Auth: 미구현
- 관광객 마이페이지: 미구현
- 결제내역: 미구현
- 예약현황: 미구현
- 맞춤코스 저장/관리: 미구현

현재 공개 네비게이션의 `마이` 버튼은 `/admin`으로 연결되어 있다. 이 구조는 플랫폼 관리자와 관광객 사용자의 역할이 섞여 보일 수 있으므로 우선 분리해야 한다.

## 2. 범위 판단

사용자 맞춤코스, 관광객 대시보드, 결제내역, 예약현황은 기존 MVP 범위를 넘어선다.

따라서 즉시 결제/예약 확정 기능을 구현하지 않고, 아래 순서로 Post-MVP 티켓을 분리한다.

1. 공개 네비의 `마이`를 `/admin`에서 분리
2. 로그인 없는 맞춤코스 임시 구성 기능
3. 관광객 계정/Auth 및 데이터 모델 설계
4. 관광객 대시보드 UI shell
5. 예약/결제 내역은 placeholder 또는 외부 연결 상태 중심으로 정의

## 3. 권장 UX 구조

### 공개 네비게이션

- `마이` 클릭 시 `/my`로 이동
- 비로그인 상태에서는 관광객용 게스트 대시보드 또는 로그인 안내 표시
- 관리자 접속은 공개 네비가 아니라 `/admin/login` 또는 운영자 전용 URL로 분리

### 맞춤코스 빌더

추천 코스를 단순히 보는 구조에서 사용자가 직접 조합하는 구조로 확장한다.

사용자가 선택할 수 있는 항목:

- 숙박: `Accommodation`
- 음식/별미/주민소득상품: `LocalIncomeProgram`
- 체험: `Experience`
- 기존 추천 코스 참고: `Course`

1차 구현은 로그인 없이 브라우저 저장소 기반으로 가능하다.

- 후보 콘텐츠 선택
- 순서 조정
- 일정 메모
- 예상 동선 확인
- 내 맞춤코스 임시 저장
- 문의하기 CTA 연결

예약 확정, 결제, 정산은 포함하지 않는다.

### 관광객 대시보드

초기 대시보드 구성:

- 맞춤코스
- 찜 목록
- 문의 내역 안내
- 예약현황 placeholder
- 결제내역 placeholder
- 알림 placeholder

계정/Auth 도입 전에는 개인정보성 데이터는 저장하지 않고, localStorage 기반의 비회원 임시 대시보드로 시작한다.

## 4. 데이터 모델 확장 후보

실제 회원 기능을 도입할 경우 필요한 모델 후보:

- `TouristUser`
- `TouristProfile`
- `SavedTripPlan`
- `SavedTripPlanItem`
- `UserNotification`
- `BookingIntent`
- `PaymentRecord`

주의:

- `BookingIntent`는 예약 확정이 아니라 문의/예약 의향 상태로 먼저 설계한다.
- `PaymentRecord`는 실제 PG 연동 승인 전까지 placeholder/read-only 모델로만 정의한다.
- 숙소·체험 운영자용 대시보드와 관광객용 대시보드는 별도 역할로 분리한다.

## 5. 티켓 편성

### T-082 공개 네비 `마이` 라우팅 분리 및 관광객 `/my` placeholder

- 공개 네비의 `마이` 링크를 `/admin`에서 `/my`로 변경한다.
- `/my` 라우트를 추가한다.
- `/my`는 관광객용 화면임을 명확히 표시한다.
- 비로그인 상태에서는 게스트 대시보드 또는 로그인 준비 중 안내를 보여준다.
- `/admin`은 운영자 전용 경로로 유지한다.
- 결제내역, 예약현황은 실제 데이터 없이 준비 중 또는 문의 기반 안내로 표시한다.
- 구현 상태: 완료
- 구현 기록:
  - `PublicNavigationShell`의 desktop/mobile `마이` 링크가 `/my`로 이동한다.
  - `/my`는 게스트 모드 관광객 대시보드 placeholder로 추가되었다.
  - 맞춤코스, 찜 목록, 문의 내역, 예약현황, 결제내역, 알림 섹션은 실제 예약/결제 확정 기능 없이 준비 상태로 표시한다.
  - `/admin`은 기존 운영자 전용 인증 경로로 유지한다.

### T-083 맞춤코스 빌더 MVP Lite

- `/course-builder` 또는 `/my/trips/new` 라우트를 추가한다.
- DB의 published 숙소, 체험, 주민소득상품을 선택 후보로 제공한다.
- 사용자가 항목을 추가/삭제/순서 변경할 수 있게 한다.
- 결과는 localStorage에 임시 저장한다.
- 맞춤코스에서 문의 CTA로 연결할 수 있게 한다.
- 예약 확정, 결제, 재고 확인은 제외한다.
- 구현 상태: 완료
- 구현 기록:
  - `/course-builder` 공개 라우트를 추가했다.
  - `/my/trips/new`는 `/course-builder`로 redirect한다.
  - `region=sowon`, `status=published` 기준 숙소, 체험, 주민소득상품을 DB에서 조회하고 DB 실패 시 fallback 데이터로 렌더링한다.
  - 사용자가 항목을 추가/삭제/순서 변경하고 여정 메모를 입력할 수 있다.
  - 결과는 `ltax_custom_trip_plan_v1` localStorage key에 저장한다.
  - 문의 CTA는 `/customer-center`로 연결하며, 예약 확정/결제/재고 확인 문구는 사용하지 않는다.

### T-084 관광객 Auth 및 사용자 데이터 모델 설계

- 관광객 로그인 도입 여부를 결정한다.
- 이메일/소셜 로그인/휴대폰 인증 중 우선순위를 정한다.
- `TouristUser`, `TouristProfile`, `SavedTripPlan`, `SavedTripPlanItem`, `UserNotification` 모델 초안을 작성한다.
- 관리자 계정과 관광객 계정의 인증 체계를 분리한다.
- 운영자/숙박관계자 대시보드는 별도 티켓으로 분리한다.
- 구현 상태: 완료
- 구현 기록:
  - 실제 Auth/DB migration은 수행하지 않고 `docs/24_TOURIST_AUTH_DATA_MODEL_SPEC.md`에 설계 초안을 분리 작성했다.
  - 관리자 세션과 관광객 세션을 분리하는 원칙을 명시했다.
  - `TouristUser`, `TouristProfile`, `SavedTripPlan`, `SavedTripPlanItem`, `UserNotification` Prisma 모델 초안을 문서화했다.
  - 개인정보 수집 최소화, localStorage 저장 제한, 로그인 후 DB 이전 기준을 정리했다.

### T-085 관광객 대시보드 1차 구현

- `/my`에 다음 섹션을 구현한다.
  - 맞춤코스
  - 찜 목록
  - 최근 본 콘텐츠
  - 문의한 항목 안내
  - 예약현황 placeholder
  - 결제내역 placeholder
  - 알림 placeholder
- Auth 미도입 상태에서는 localStorage 기반으로 표시한다.
- Auth 도입 후 DB 기반으로 전환 가능하도록 컴포넌트를 분리한다.

### T-086 예약현황/결제내역 정책 결정

- 실제 예약/결제 기능을 도입할지, 외부 예약 링크 상태 표시만 할지 결정한다.
- MVP 원칙상 결제, 정산, 실시간 예약 확정은 기존 범위 밖이다.
- 도입 시 필요한 항목:
  - PG
  - 환불/취소 정책
  - 숙소/체험 재고
  - 정산
  - 운영자별 권한
  - 고객 개인정보 보관 정책
- 결정 전까지 `/my`에서는 "문의/외부 예약 연결 내역" 수준으로만 표현한다.

### T-087 관광객 알림 센터 설계

- 알림 유형을 정의한다.
  - 문의 접수 안내
  - 맞춤코스 저장 안내
  - 이벤트/혜택 알림
  - 운영자 답변 알림
- 실제 push/email/SMS 발송은 제외하고 화면 내 알림 placeholder부터 구현한다.
- 개인정보 동의와 마케팅 수신 동의는 별도 정책으로 분리한다.

## 8. 구현 주의사항

- 관광객용 `/my`와 운영자용 `/admin`은 절대 혼용하지 않는다.
- 결제내역/예약현황은 실제 결제·예약 시스템이 없으면 placeholder로만 표현한다.
- "예약 완료", "결제 완료" 같은 확정 문구는 실제 기능 도입 전 사용하지 않는다.
- 맞춤코스는 `Course` 원본 모델을 변경하지 않고, 사용자 저장 모델 또는 localStorage 구조로 별도 관리한다.
- `LocalIncomeProgram`은 음식/별미/주민소득상품 선택지로 유지하고 `Experience`에 흡수하지 않는다.
